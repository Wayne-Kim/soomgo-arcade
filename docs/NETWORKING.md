# 네트워킹 — 하이브리드 근거리 멀티플레이어

숨고 아케이드는 사용자가 원하는 "쉬운 근거리 연결" 경험을 제공하는 동시에, 실시간 데이터 전송 속도를 아케이드 게임에 적합하게 빠르고 안정적으로 유지할 수 있는 **하이브리드** 모델을 사용하여 **4~8인 근거리 멀티플레이어**를 지원합니다.

## 왜 데이터 채널로 블루투스를 사용하지 않나요?

실시간 데이터 채널로 블루투스를 사용하는 것은 빠른 격자판 아케이드 게임에 적합하지 않습니다:

- **대기 시간(Latency)** — 클래식 링크는 일반적으로 약 250~300ms의 대기 시간을 추가하며, 이는 본 게임의 입력 버퍼 예산을 초과합니다.
- **기기 제한** — 피코넷(Piconet)의 한계는 약 7대이며 (실제로는 약 4대), 이는 본 게임의 목표인 8인 플레이에 미치지 못합니다.
- **Godot 네이티브 지원 부재** — Godot 엔진에서 공식적으로 지원하는 실시간 블루투스 전송 프로토콜이 없습니다.

따라서 블루투스는 데이터 전송 경로로 사용되지 **않습니다**. 대신 블루투스가 실제로 유용한 영역인 **기기 검색, 페어링, 쉬운 초대 및 연결 실패 시의 폴백(Fallback) 수단**으로만 사용됩니다.

## 전송 레이어 역할 정의 (단일 진실 공급원)

| 관심사 | 채널 | 역할 |
| --- | --- | --- |
| 실시간 입력 및 상태 | **Godot ENet** (데스크톱 LAN/Wi‑Fi, 모바일 Wi‑Fi Direct) | 데이터 채널. 결정론적 록스텝(Deterministic Lockstep) 방식 사용. |
| 동일 네트워크 내 검색 | **LAN UDP 비콘 (Beacon)** | 호스트가 ENet 엔드포인트를 브로드캐스트하여 클라이언트가 자동 연결되도록 함. |
| 페어링 / 초대 / 폴백 | **블루투스 / Wi‑Fi 핫스팟** | 기기들을 하나의 무선 네트워크로 연결시킵니다. **게임 데이터는 절대 전송하지 않습니다.** |

이 역할 정의는 `scripts/net/ConnectionFlow.gd` 및 `scripts/net/NetBudget.gd`에 반영되어 있습니다.

## 연결 흐름 (승인 기준 1)

`ConnectionFlow`는 소켓 연결이 필요 없는 결정론적 상태 머신으로, 연결(Connect) 화면이 이 머신에 바인딩됩니다:

```
검색 시작(begin_scan) ─▶ SCANNING ──(동일 네트워크의 호스트 발견)──▶ CONNECTING ──▶ CONNECTED
                             │
                             └──(검색 시간 초과 / 호스트 없음)──▶ FALLBACK ──▶ (블루투스로 페어링
                                                                               또는 핫스팟 활성화)
                                                                                    │
                                                                             폴백 네트워크 준비 완료
                                                                                    ▼
                                                                                재검색(re-SCAN)
```

- **동일 무선 네트워크 → 자동 ENet.** `LanBeacon`(`NetBudget.BEACON_PORT` 포트의 UDP 브로드캐스트)을 통해 클라이언트는 수동으로 IP를 입력할 필요 없이 호스트의 ENet 엔드포인트를 찾아 자동으로 연결합니다.
- **다른 네트워크인 경우 → 폴백 안내.** 연결 화면에서 먼저 Wi‑Fi 핫스팟 생성을 권장합니다 (피코넷보다 대역폭이 넓고 연결 기기 제한이 적음). 그 대안으로 가까운 거리에서 쉽게 페어링할 수 있는 블루투스 페어링 안내를 제공합니다. 공통 네트워크가 구성되면 재검색을 통해 호스트를 발견하고 자동으로 연결합니다.

## 넷코드 모델 및 대기 시간 예산 (승인 기준 2)

핵심 시뮬레이션 코드(`scripts/core/`)는 이미 **결정론적**(고정된 `TICK_DELTA` + 시드 기반 RNG)으로 설계되어 있으므로, 네트워킹은 **결정론적 록스텝(Deterministic Lockstep)** 방식을 따릅니다: 호스트가 플레이어 ID와 매치 시드를 부여하면, 모든 피어(Peer)가 틱별 입력을 중계하고, 모든 피어가 완전히 동일한 `Simulation.step()`을 실행합니다.

대기 시간 예산은 `scripts/net/NetBudget.gd`에 정의되어 있습니다:

| 항목 | 값 |
| --- | --- |
| 목표 입력 왕복 시간 (RTT) | **≤ 100 ms** (`TARGET_RTT_MS`) |
| 네트워크 품질 경고 기준 | 60 ms (`WARN_RTT_MS`) |
| 입력 지연 (Input Delay) | 3 틱 ≈ 50 ms (`INPUT_DELAY_TICKS`) |
| 최대 플레이어 수 | 8명 (`MAX_PLAYERS`) |

동일 네트워크 LAN/Wi‑Fi 연결 기준 예산 세부 정보:

```
입력 샘플링          ~1 틱 (~16.7 ms)
편도 전송            ~10–25 ms
호스트 중계 및 회신    ~10–25 ms
-----------------------------------
목표 입력 왕복 시간   ≤ 100 ms
```

짧은 고정 **입력 지연**을 적용하여 롤백(Rollback) 없이 단방향 대기 시간을 감추고, 8명의 피어 환경에서도 움직임이 부드럽게 유지되도록 합니다. 연결 화면은 실시간으로 측정된 RTT를 바탕으로 품질을 양호/보통/느림 단계로 분류하여 사용자에게 표시합니다 (`NET_QUALITY_*`, `NET_RTT`).

**검증:** `tests/net_loopback.gd` 테스트는 실제 ENet 호스트와 클라이언트를 구동하여 PING/PONG 왕복 시간을 측정하고, 이 값이 `TARGET_RTT_MS` 이내인지 확인합니다 (루프백은 보통 ~10–15 ms 발생). 또한 동일한 입력을 제공받은 두 개의 시드 기반 시뮬레이션이 정확히 동일한 상태에 도달하는지 검증함으로써 록스텝의 일관성을 증명합니다.

## 모듈 구성

- `scripts/net/NetBudget.gd` — 대기 시간 예산, 프로토콜 버전 및 포트 관리 (SSOT).
- `scripts/net/NetProtocol.gd` — 경량 바이너리 통신 코덱 (HELLO/WELCOME/INPUT/PING/PONG/BEACON/START/DROP).
- `scripts/net/LanBeacon.gd` — LAN 탐색 비콘 (UDP 브로드캐스트 광고/수신).
- `scripts/net/ConnectionFlow.gd` — 연결 및 폴백 전환 결정을 처리하는 상태 머신.
- `scripts/net/NetSession.gd` — ENet 호스트 개설 및 참여, 입력 중계, RTT 핑 측정, 매치 시작(START) 및 이탈(DROP) 처리.
- `scripts/net/NetMatch.gd` — *시작된* 매치의 결정론적 록스텝 구동기 (틱당 입력 지연 적용, 가장 느린 피어 대기, 지연 한도 기반 대기 + 마지막 입력 예측, 로컬 봇 처리).
- `scripts/net/NetContext.gd` — 연결 화면에서 실행 중인 `NetSession`을 게임 씬으로 전달하는 매개체.
- `scripts/ui/Connect.gd` + `scenes/Connect.tscn` — 연결/페어링 화면 및 호스트용 **매치 시작** 버튼.

## 네트워크 매치 시작 및 실행

피어들의 연결이 완료되면 **호스트**가 연결 화면에서 **매치 시작** 버튼을 누릅니다:

1. `NetSession.start_match()`가 **ID 인덱스 기반의 로스터**를 생성하고(연결된 실제 플레이어에게 ID를 할당하고, 나머지 빈 슬롯이나 `Spec.MIN_PLAYERS`까지의 패딩 영역은 봇으로 채움), 공유 시드와 로스터 정보가 담긴 **START** 메시지를 브로드캐스트합니다. 이를 수신한 모든 피어는 **바이너리 수준에서 동일한 `Simulation`**을 구축합니다. 최소 인원 기준을 충족하지 못하면 매치 시작이 차단되며, START 이후에 뒤늦게 들어오는 HELLO 요청은 차단됩니다 (호스트는 `started` 상태가 되면 참가 승인을 중단함).
2. 모든 피어는 세션 인스턴스를 `NetContext`에 넘겨주고 `Game.tscn` 씬으로 진입합니다. `Game.gd`는 키보드 입력을 받아 `NetSession.local_player_id`로 구동하며, 매 틱마다 입력을 **브로드캐스트**하는 동시에 다른 피어들의 입력을 `NetMatch`를 거쳐 `Simulation.set_input`에 적용합니다. 이후 모든 라이브 피어의 해당 틱 입력이 확보되었을 때만 고정 틱 단위로 `Simulation.step`을 전진시킵니다 (네트워크 지연 발생 시 싱크가 어긋나는 대신 가장 느린 피어를 기다림). 빈 슬롯의 봇 AI는 공유 시드를 기반으로 각 피어의 로컬에서 개별 실행되므로, 패킷 전송 없이도 완전히 동일한 동작을 수행하여 동기화가 유지됩니다.
3. 호스트는 **중계 허브** 역할을 수행합니다. 클라이언트-서버 구조의 ENet에서는 클라이언트 패킷이 호스트에게만 도달하므로, 호스트는 수신한 클라이언트 INPUT을 다른 모든 클라이언트 피어들에게 재전송해 줍니다.
4. **피어 이탈 처리:** 호스트가 **DROP** 메시지를 브로드캐스트하면, 모든 피어는 이탈한 플레이어의 입력 상태를 마지막으로 수신된 프레임 상태로 고정(repeat-last prediction)합니다. 이 예측 메커니즘은 타이밍에 무관하게 작동하므로, DROP 메시지를 통해 이탈을 즉시 감지한 피어와 지연 시간 한계(stall budget) 초과로 늦게 감지한 피어 모두 완전히 동일한 시뮬레이션 상태에 도달합니다. 라운드는 멈춤이나 동기화 오류 없이 계속 진행되며 화면에 비선점형 경고 메시지가 표시됩니다.

이 모든 과정은 헤드리스 모드로 검증됩니다: `tests/net_loopback.gd`는 ENet을 통해 실제 매치를 시작하고 매 틱마다 두 피어의 시뮬레이션 해시값이 정확히 일치하는지 확인합니다. `tests/run_net_tests.gd`는 NetMatch 결정론 및 이탈 복구 메커니즘을 테스트합니다. `tests/smoke_ui.gd`는 호스트의 매치 시작 상태 및 네트워크가 연동된 게임 씬 구동을 확인합니다.

## 연결 화면 UI (승인 기준 3)

본 저장소에는 디자인 SSOT가 정의되어 있지 않으므로, 연결 화면 역시 로비 화면과 동일한 접근성 테마(`assets/theme.tres`) 및 공통 UX 규칙을 적용합니다:

- **지원 상태:** 검색 중(scanning), 검색 결과 없음(empty), 연결 중(loading), 오류 발생(error), 작업 중 상태 비활성화(disabled), **호스트 매치 시작 대기 중**, **매치 시작 중**(로딩 오버레이), 그리고 모든 컨트롤에 대한 명확한 포커스 링 표시.
- **접근성:** 호스트용 **매치 시작** 버튼을 포함한 모든 인터랙티브 컨트롤은 접근성 레이블/설명(`A11y.gd`)이 제공되며, 포커스 이동이 가능하고 고대비 테마가 적용되어 있습니다.
- **다국어(i18n):** 사용자에게 표시되는 모든 문자열은 `localization/ui_strings.csv` → `tr("KEY")` 방식으로 외부화되어 있습니다. 다국어 개수가 하드코딩되지 않아 코드 수정 없이 CSV 열을 추가하는 방식으로 새로운 언어를 확장할 수 있습니다.

## 범위 예외 및 제약 사항

본 구현 작업은 **전송 레이어 + 연결/페어링 UI + 대기 시간 예산** 설정을 제공하며, 이를 실제 **시작된 매치**에 연결하는 데 집중합니다 (호스트가 매치를 시작하면 모든 피어가 동일한 결정론적 게임을 플레이함). 플랫폼 단위의 블루투스/Wi-Fi Direct 페어링은 OS 수준의 기능이므로 여기서는 탐색 및 폴백을 안내하는 가이드 경로로만 노출되며 (실제 데이터 채널은 ENet 고정), 특정 플랫폼 전용 페어링 SDK 연동은 범위에서 제외되고 `ConnectionFlow` 상태 머신의 폴백 API 내부로 격리됩니다.
글로벌 매치메이킹 서버, 인터넷 및 릴레이 중계 플레이, 관전자 기능, 게임 내 채팅 및 튕긴 플레이어의 게임 도중 재접속 기능은 지원 범위에 포함되지 않습니다.
